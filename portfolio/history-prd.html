FEATURE PRD

历史记录与结果复盘功能 PRD

历史记录不是一个“附属列表”，而是 AI 产品从单次生成走向持续优化的基础设施。该功能让用户能找回、对比和复盘润色结果，也让产品侧沉淀输入、输出、模型、Prompt 版本和 Bad Case 数据，为后续多模型评测和用户体系打基础。

目录 一、功能背景 二、用户场景 三、页面入口与信息架构 四、字段设计 五、核心交互流程 六、状态流转 七、异常与兜底 八、数据埋点与指标 九、后续迭代 十、验收标准

SECTION 01

一、功能背景

用户使用论文润色工具时，往往不是一次性完成任务，而是会多次输入、对比结果、复制片段、回看之前版本。如果没有历史记录，用户只能依赖浏览器当前状态，一旦刷新或关闭页面，结果就会丢失。对于产品侧来说，没有历史数据也无法复盘模型效果、定位 Bad Case 或评估 Prompt 版本。

用户价值

支持找回历史润色结果、对比不同版本、继续修改论文段落，降低重复操作成本。

产品价值

沉淀任务数据，为效果评估、Prompt 迭代、模型对比和用户行为分析提供依据。

技术价值

建立任务状态表和结果存储结构，为后续登录体系、额度体系和多租户数据隔离打基础。

SECTION 02

二、用户场景

用户 场景 需求 功能响应
本科毕业生 多次润色毕业论文段落。 希望找回之前比较满意的表达。 按时间展示历史任务，点击后恢复原文和结果。
研究生 对实验结果段落反复修改。 希望确认数值和结论没有被 AI 改错。 展示原文、润色后文本、修改说明和模型版本。
产品评测者 批量测试模型在不同用例上的表现。 需要记录成功、失败、低分用例。 保存任务状态、错误信息、评分和 Bad Case 标签。
后续注册用户 跨设备查看自己的润色记录。 希望历史记录与账号绑定。 匿名历史迁移到用户账号，支持分页和搜索。

SECTION 03

三、页面入口与信息架构

V1 阶段采用“主工作台右侧历史面板 + 历史详情页规划”的轻量结构，先满足快速回看和结果恢复。

  1. 主工作台右侧显示最近 10 条任务，包含状态、时间、原文摘要和模型信息。
  2. 点击某条历史记录后，将原文、润色结果、修改说明和剩余问题恢复到主结果区。
  3. 失败任务展示错误状态和重试入口，不覆盖用户原始输入。
  4. 后续独立历史页支持搜索、筛选、收藏、评分和导出。

SECTION 04

四、字段设计

字段设计需要同时服务用户回看、问题排查和后续评测分析。V1 匿名阶段不采集姓名、手机号等身份信息。

字段 类型 说明 是否 V1 必需
task_id string 任务唯一编号，用于前后端查询和问题定位。 P0
input_text text 用户输入的原始论文段落。 P0
polished_text text 模型返回的润色后文本。 P0
result_json json 包含修改说明、剩余问题、知识点等结构化结果。 P0
status enum running、success、failed 三种任务状态。 P0
model_provider string 模型服务提供方，如 modelgate。 P0
model_name string 实际调用的模型 ID，用于效果复盘和成本统计。 P0
prompt_version string Prompt 版本号，用于后续 A/B 测试和回归评测。 P1
score number 人工或自动评估分数。 P1
badcase_tag string 低分结果的问题类型，如过度扩写、事实错误、格式失败。 P1
created_at datetime 任务创建时间。 P0

SECTION 05

五、核心交互流程

5.1 新建润色任务

  1. 用户输入论文段落，选择润色目标并点击“开始润色”。
  2. 前端调用后端接口，后端创建 running 状态任务。
  3. 后端调用模型并解析结构化输出。
  4. 调用成功后更新为 success，并保存结果字段。
  5. 前端展示结果，同时刷新右侧历史记录。

5.2 查看历史任务

  1. 用户点击历史记录中的任务卡片。
  2. 前端根据 task_id 获取任务详情。
  3. 页面恢复原文、润色结果、修改说明、剩余问题和知识点。
  4. 如果任务为 failed，则展示失败原因和重新润色入口。

5.3 复盘低分结果

  1. 评测者在评估页面标记低分任务。
  2. 为任务添加 Bad Case 标签和问题说明。
  3. 将该任务沉淀为回归测试用例。
  4. Prompt 或模型变更后重新运行该用例，比较结果变化。

SECTION 06

六、状态流转

状态 触发条件 页面展示 后续动作
running 用户提交任务，后端已创建记录。 显示生成中，不允许重复提交同一任务。 等待模型返回或超时。
success 模型返回并成功解析结构化结果。 展示润色结果、修改说明和历史卡片。 允许复制、查看详情、加入评测。
failed 模型调用失败、超时、JSON 解析失败或数据库写入失败。 展示失败提示，保留用户原文。 允许重试，记录错误类型。

SECTION 07

七、异常与兜底

异常场景 用户提示 系统处理
输入为空 请先输入需要润色的论文段落。 不创建任务，不调用模型。
输入过短 当前文本较短，建议补充上下文以获得更可靠的润色结果。 允许继续，但在结果中提示信息不足风险。
模型调用失败 模型服务暂时不可用，请稍后重试。 任务记为 failed，保存错误类型。
结构化解析失败 结果格式异常，请重新生成。 保留原始输出用于排查，前端展示兜底文案。
历史记录加载失败 历史记录暂时无法加载，不影响继续润色。 主流程可继续，记录接口错误。

SECTION 08

八、数据埋点与指标

history_view：用户是否查看历史记录。
history_restore：用户是否点击历史任务恢复结果。
copy_result：用户是否复制润色结果。
retry_failed_task：失败任务是否被重试。
task_success_rate：任务成功率。
badcase_mark_rate：低分用例标记比例。
指标解读：如果历史记录点击率高，说明用户有复盘需求；如果复制率高，说明结果有实际使用价值；如果 failed 重试率高，说明失败兜底文案和模型稳定性需要重点优化。

SECTION 09

九、后续迭代

V1.0 匿名历史记录

保存最近任务，支持查看、恢复、失败提示和基础状态展示。

V1.1 历史详情与筛选

支持搜索、筛选成功/失败、查看模型和 Prompt 版本、手动评分和 Bad Case 标签。

V1.2 登录与数据迁移

支持用户登录、匿名历史迁移、收藏、导出、额度统计和跨设备同步。

V1.3 模型评测闭环

将历史任务接入多模型对比和 Prompt 回归测试，形成质量看板。

SECTION 10

十、验收标准

任务可追踪

每次润色任务都有唯一编号、任务状态、创建时间、输入和输出记录。

结果可恢复

用户点击历史任务后，页面能恢复原文、润色结果和结构化说明。

失败可解释

模型失败、解析失败和历史加载失败都有明确提示，不丢失用户输入。

数据可复盘

记录模型、Prompt 版本和错误类型，为评测和 Bad Case 分析提供依据。

隐私可控

匿名阶段不采集个人身份信息，后续登录版本再设计用户数据隔离。

入口清晰

主工作台能直接看到最近任务，独立历史页规划支持后续扩展。
