POSITIONING

个人定位

我希望成为能理解业务、懂 AI 链路、能推动产品落地的 AI 产品经理。过往经历中，我参与过保险业务 RAG 智能问答系统的测试与效果评估，也独立完成了论文润色 Agent 的前后端 Demo、模型接入、数据库记录和 Docker 部署。

PRODUCT

产品背景与用户痛点

论文写作用户并不是单纯想让 AI “改得更好看”，而是希望获得更正式的表达、理解修改原因，并能追踪多次修改结果。

表达不够学术

原文常出现口语化、主观化或表达冗余的问题，需要转换为更客观、凝练的论文表达。

修改不可解释

通用聊天工具通常只给结果，用户不知道 AI 改了哪里，也难以判断是否保留了原意。

结果难复盘

多次润色后结果散落在对话中，缺少任务记录、版本对比和后续质量评估基础。

WORKFLOW

核心产品流程

第一阶段优先验证核心闭环：输入文本、选择润色需求、生成结构化结果、保存匿名历史记录。这个选择降低了使用门槛，也为后续登录、质量评估和数据闭环打基础。

  1. 用户输入论文段落
  2. 选择学术化润色、逻辑优化等需求
  3. 后端创建 running 任务记录
  4. 调用大模型返回结构化结果
  5. 保存 success / failed 状态与结果
  6. 前端展示历史记录，可点击恢复结果

AI ENGINEERING

Prompt 设计与模型选择

设计点 产品考虑
角色约束 让模型扮演学术论文润色助手，输出更贴近论文语境。
任务约束 不只润色，还要解释修改原因、指出剩余问题和学习建议。
结构约束 要求固定 JSON 字段，方便前端展示、数据库记录和质量评估。
事实约束 强调不新增事实、不改变论文结论，降低 AI 事实改写风险。
第一阶段选择 OpenAI 兼容模型，主要看重中文理解、指令遵循、结构化输出和稳定性。系统通过 ModelGate 接入，后续可以对比 Qwen、DeepSeek、Claude 等模型，避免与单一模型强绑定。

MODEL SELECTION

模型对比方案

模型不是只看“回答好不好看”，而要拆成可评估指标。

40%润色质量
25%语义保持
15%结构化输出
10%响应速度
10%调用成本

EVALUATION

测试集与效果评估

我会用 50 条左右的小测试集覆盖典型输入、边界输入和风险输入。

典型文本

口语化表达、学术表达不足、长句逻辑混乱、术语密集文本。

边界输入

空输入、超短输入、超长输入、格式混乱、包含表格或编号的文本。

风险输入

包含事实数据、实验结论、Prompt 注入、敏感内容和可能违规请求。

FALLBACK

兜底策略与异常处理

异常情况 兜底方案
模型超时 前端提示重试，后端记录 failed 状态，便于后续分析失败率。
JSON 格式错误 自动进行一次结构修复重试，仍失败则返回可读错误。
润色结果过短 提示结果可能不完整，允许重新生成或换模型。
输入过长 分段处理或提示用户拆分提交，避免上下文截断。
数据库写入失败 优先返回润色结果，同时提示历史保存失败。

COMPETITION

竞品分析与定位

Paperpal

偏学术写作，适合借鉴投稿导向和论文场景专业化能力。

Grammarly

强在跨平台写作辅助，说明写作工具要融入用户工作流。

秘塔写作猫

中文写作辅助能力完整，说明本土化表达和中文场景很重要。

QuillBot

强在改写和 paraphrase，启发多种风格和改写强度设计。

ChatGPT

自由度高，但专用产品要靠结构化流程和稳定体验形成差异。

我的差异化定位：聚焦中文论文段落润色，强调可解释润色、历史追踪和学习反馈，而不是做大而全的写作助手。

INTERVIEW SUMMARY

面试总结表达

这个项目让我完整经历了一个 AI 产品从想法到 Demo 落地的过程。我不仅关注模型能否生成结果，也关注输入方式、Prompt 结构、输出展示、异常兜底、任务记录和后续数据闭环。我理解 AI 产品经理的价值不是简单接入模型，而是把模型能力嵌入真实场景，设计稳定、可解释、可评估、可持续迭代的产品体验。
