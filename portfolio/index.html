PRODUCT

产品背景与用户痛点

论文写作用户并不是单纯想让 AI “改得更好看”，而是希望获得更正式的表达、理解修改原因，并能追踪多次修改结果。

表达不够学术

原文常出现口语化、主观化或表达冗余的问题，需要转换为更客观、凝练的论文表达。

修改不可解释

通用聊天工具通常只给结果，用户不知道 AI 改了哪里，也难以判断是否保留了原意。

结果难复盘

多次润色后结果散落在对话中，缺少任务记录、版本对比和后续质量评估基础。

WORKFLOW

当前 Demo 的真实工作流

第一阶段优先验证核心闭环：输入文本、选择润色需求、生成结构化结果、保存匿名历史记录。这个选择降低了使用门槛，也为后续登录、质量评估和数据闭环打基础。

  1. 用户输入论文段落
  2. 选择学术化润色、逻辑优化等需求
  3. 后端创建 running 任务记录
  4. 调用大模型返回结构化结果
  5. 保存 success / failed 状态与结果
  6. 前端展示历史记录，可点击恢复结果

Prompt 设计与模型选择

Prompt 约束和模型接入已经实现；系统化多模型对比还没有做完，所以这里明确写成“选择依据”和“后续对比计划”。

设计点 状态 产品考虑
角色约束 已实现 让模型扮演学术论文润色助手，输出更贴近论文语境。
任务约束 已实现 不只润色，还要解释修改原因、指出剩余问题和学习建议。
结构约束 已实现 要求固定 JSON 字段，方便前端展示和数据库记录。
事实约束 待加强 结合事实敏感测试用例，强化“不新增事实、不改变结论”的校验。
第一阶段选择 OpenAI 兼容模型，是因为它在中文理解、指令遵循和结构化输出上更适合作为 Demo 验证模型。后续会用同一套测试集对比 Qwen、DeepSeek、Claude 等模型。

COMPETITION

竞品分析与定位

这部分是基于同类产品功能观察提炼的产品定位，不包装成完整商业竞品报告。

Paperpal

偏学术写作，适合借鉴投稿导向和论文场景专业化能力。

Grammarly

强在跨平台写作辅助，说明写作工具要融入用户工作流。

秘塔写作猫

中文写作辅助能力完整，说明本土化表达和中文场景很重要。

QuillBot

强在改写和 paraphrase，启发多种风格和改写强度设计。

ChatGPT

自由度高，但专用产品要靠结构化流程和稳定体验形成差异。

我的差异化定位：当前 Demo 聚焦中文论文段落润色，强调可解释润色、历史追踪和学习反馈。未来如果继续做，需要通过更系统的竞品体验和用户访谈验证这个定位。
